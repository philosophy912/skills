cat-video-analyzer 工作流程 分析小米摄像头录制的猫咪视频，识别 狸花猫 tabby 黑猫 black 在猫砂盆 / 喂食机区域的 吃饭 上厕所 行为，按日输出 Markdown 报告

视频来自小米摄像头（一个拍猫砂盆、一个拍喂食机，每段约 1 分钟），存放在 NAS 上。 流程不是“一键脚本”——中间的逐帧识别由 agent 的多模态能力完成，脚本只做确定性工作 （扫描、抽帧、去重、聚合、报告）。这样 skill 不绑死任何模型 API，能在任何“能读图 + 能跑 bash”的 agent 上工作。

约定：下文 CATVA 代表 python3 "${CLAUDE_PLUGIN_ROOT}/skills/cat-video-analyzer/scripts/catva.py"。 所有子命令通过 stdout 输出 JSON，方便 agent 解析。Windows 上若 python3 不存在，改用 python。

实施状态

本文档描述的流程包含四个 token 优化杠杆。下表标注每个杠杆的落地状态—— ✅ 已实现 表示当前代码已支持，✅ 已实现 表示已设计但尚未实现。 文档反映的流程已全部落地实现。

  • 本地运动检测预筛（新增 prefilter 子命令，砍掉无运动段）✅ 已实现
  • 图像降采样（ffmpeg scale 到长边 768，配置项 frame_max_side）✅ 已实现
  • 采样间隔按 context 分段（猫砂盆 6s / 喂食机 12s，救黑猫快速进出）✅ 已实现
  • 段内 dHash 去重（相邻相似帧合并为代表帧）✅ 已实现
四个杠杆现已全部落地。降采样后代表帧长边 ≤768（单帧 token ~440）， 预筛砍掉 85%+ 无运动段——从原始每段必送 1 张 1080p 大图的烧爆状态，压到 ~40 万 token/天。

token 消耗账（被藏起来的关键账）

小米摄像头连续录像、NAS 上每分钟一个文件，意味着每天固定 2880 段（24h × 2 摄像头 ÷ 1min）。 多模态识别的 token 取决于两个独立维度：送多少帧 × 每帧多大。 本文档原先只谈前者（帧数），对后者（分辨率）只字未提——而分辨率才是 6–9 倍的杠杆。

单帧 token ≈ 宽 × 高 / 750（Claude 视觉）。一张 1920×1080 ≈ 2700 token， 降采样到长边 768（768×432）≈ 440 token，差 6 倍。

方案段/天代表帧/天单帧 token总 token/天vs 原始
原始（当前代码）2880~2880（每段≥1）~2700 (1080p)~777 万1×
+ 预筛~300 (15%)~750~2700~200 万3.9×

从一天近千万 token、上百美金，压到 ~40 万 token、个位数美金。注意：dHash 去重（已实现的那项）省的是帧数， 在这张表里几乎不占行——它无法独立解决连续录像的量级问题，必须和预筛、降采样配合。

流程详解

1

scan · 扫描当天待处理文件

执行者：scripts · agent 仅发起调用
CATVA scan --date 2026-06-11

递归扫描 nas.litter_box / nas.feeder，从文件名解析时间戳（支持 20260611_143000、2026-06-11_14-30-00 等，失败回退 mtime）， 按 --date 过滤；读 state.json 做增量去重，只返回指纹 (mtime, size) 不匹配的文件。

输出 JSON：todo 为当天且尚未处理的视频清单（每项含 path 和 context），skipped 为已处理被跳过的数量。 --force 可忽略去重全量重列。

todo 为空且 skipped > 0 → 当天都处理过了，直接跳到第 5 步出报告。
▼
1.5

prefilter · 运动检测预筛 ✅ 已实现

执行者：scripts · ffmpeg + Pillow · 纯本地，不送模型
CATVA prefilter --date 2026-06-11
# 调阈值后只复检上轮判为无运动的段：
CATVA prefilter --date 2026-06-11 --recheck-silent

连续录像下省 token 的最大杠杆。对 todo 里每个视频，用 ffmpeg 极低帧率抽 3 帧（首 0s / 中 30s / 尾 60s），缩到 32×32 灰度，算相邻帧平均像素差。

  • 差值 ≤ 阈值 → 判定「整段静止」不进识别管线、不写 jsonl，但在 state.json 标记 silent（否则下次 scan 又重列，预筛白跑）
  • 差值 > 阈值 → 进「有运动」清单，交给后续 extract + 识别

用相邻帧差而非对背景帧差：光照渐变是低频全局变化，相邻 30s 几乎归零；猫的位移是高频局部变化仍能捕获——故红外夜视渐变不会把空场景误判成有运动。

输出 JSON：motion_videos（有运动段清单）、skipped_silent（静默跳过数）。典型连续录像下静默跳过 85%+ 的段——这是从 2880 段/天 砍到 ~300 段/天 的关键。
漏检边界：3 帧采样间隔 30s，理论上漏 < 30s 的瞬时活动。但猫吃饭/上厕所单次 ≥ 1–2 分钟，且聚合阶段 30s 间隔容忍会补回。--recheck-silent 只重判上轮 silent 段，调阈值时不影响已识别的运动段。
▼
2

extract · 抽帧 + 降采样 + 去重（输出代表帧）

执行者：scripts · ffmpeg + Pillow
# 单个视频
CATVA extract --video /path/to/clip.mp4 --context litter_box --date 2026-06-11

# 一次并行处理当天所有视频（更快）
CATVA batch-extract --date 2026-06-11

调 ffmpeg 的 fps=1/interval 滤镜均匀抽帧， frame_ts = 视频起始时间 + 帧偏移，由脚本算好，agent 不用自己算。

降采样 ✅ 已实现

ffmpeg -vf 同时挂 scale='min(768,iw)':-2，抽出的代表帧长边 ≤ 768px。 单帧 token 从 ~2700（1080p）降到 ~440——这是除预筛外的第二大杠杆（6 倍）。尺寸由配置项 processing.frame_max_side 控制（默认 768）。768 在 1 tile 内、留足识别虎斑条纹 与「鼻尖靠近食碗」的证据细节；不建议降到 480p，会牺牲识别精度。

采样间隔按 context 分段 ✅ 已实现

配置项 frame_interval_seconds 从单一值改为按 context 分段表：

[processing]
frame_interval_seconds.litter_box = 6   # 猫砂盆：救黑猫 <24s 快速进出
frame_interval_seconds.feeder     = 12  # 喂食机：吃饭是长活动，12s 足够

向后兼容：读到单一整数则两个 context 都用它（老配置不破）。6s 间隔下猫砂盆活跃段抽 10 帧， <24s 的快速进出能命中 3–4 帧而非 0–1 帧，配合 min_frames 降到 1（见聚合规则）消除结构性漏检。

相邻、画面几乎相同的帧会被合并（dHash 汉明距离 ≤ 阈值），每组只留一个代表帧， 附带 time_range（覆盖时段）和 represents（代表了几帧）。

⚠️ 去重省的是帧数，不是每帧 token。在预筛+降采样落地前，dedup 是唯一省 token 的手段，但对连续录像的量级问题杯水车薪——见上方 token 账。
去重需 Pillow（pip install pillow）。未装时输出全部帧（无 time_range），流程不变，只是不省 token。可用 CATVA doctor 检查。
▼
3

识别每张代表帧 ⬅ 核心步骤

执行者：agent · 自身多模态读图能力 · 不调脚本不调 API

对返回的每个代表帧，用读图能力打开 frame_path，按 references/recognition-protocol.md 判断：

  • 有哪些猫：tabby 虎斑条纹 + 额头 M 花纹 · black 全身黑无花纹 · 无法确定填 unknown · 无猫返回 []
  • 在做什么：eating toileting idle absent（仅当画面完全无猫）
context算 toileting算 eating否则
litter_box 猫砂盆刨沙 / 蹲伏砂盆内 / 进出砂盆不可能 → idleidle（路过/休息）/ absent
feeder 喂食机不可能 → idle低头 / 鼻尖嘴靠近食碗出粮口 / 咀嚼idle / absent
路径级硬约束：猫砂盆画面里不会真正进食，喂食机画面里不会真正如厕。识别时就应正确归类（ingest 阶段还有代码兜底）。

对每帧产出一个 JSON 对象（带 time_range 的要原样回传， ingest 会按采样间隔扩展为整段）：activity.evidence 必须填一句话引用画面细节， 它逼你看图而非猜，也会写进最终报告供主人抽查。

批量识别省往返：一个视频通常只剩 2-3 个代表帧，建议一次打开全部代表帧、一次性输出 JSON 数组。
▼
4

ingest · 把识别结果写回

执行者：scripts · 校验 + 扩展 + 落盘 + 标记
CATVA ingest \
  --video /path/to/clip.mp4 \
  --date 2026-06-11 \
  --results '[{"frame_ts":"...","context":"litter_box","cats":[{"identity":"black","confidence":0.9}],"activity":{"type":"toileting","evidence":"..."},"time_range":["...","..."]}]'

--results 也支持 @文件路径。ingest 会：

  1. 校验每条结果字段合法性（schema.validate_frame_result）
  2. 应用路径级硬约束（猫砂盆的 eating → idle，反之亦然）
  3. 带 time_range 的结果按采样间隔扩展为整段内多个时间点写入 raw/日期.jsonl
  4. 把该视频标记为已处理（mark_processed）—— 只有这步成功才标记

对 motion_videos（预筛后）里每个文件重复第 2–4 步（或用 batch-extract 一次抽完，再逐个 ingest）。未启用预筛时退化为对 todo 全量处理。

▼
5

report · 生成报告

执行者：scripts · 从 jsonl 重聚合，零识别开销
CATVA report --date 2026-06-11

读 raw/日期.jsonl 里当天全部帧（跨视频文件全局聚合）， 生成 reports_dir/2026-06-11.md。报告含：汇总（吃饭/上厕所总次数）、 每只猫小节（tabby → black → unknown 固定顺序）、按时间排列的事件时间线（含证据摘要）。

报告是 jsonl 的纯函数渲染。改了报告格式想重出？直接再跑 report，不消耗任何识别（读图）开销，秒出。

持久化数据布局

~/.config/cat-video-analyzer/ (Windows: %APPDATA%\cat-video-analyzer\) ├── config.toml 用户配置（NAS 目录、去重参数、时区） ├── state.json 已处理文件指纹（增量去重核心） ├── state.json.lock 文件锁哨兵 └── raw/ └── 2026-06-11.jsonl 当天所有帧的识别结果（每行一帧 JSON）
  • state.json 只存指纹 → 体积小、加载快。scan 用它跳过已识别文件。
  • raw/*.jsonl 存逐帧结果 → 按日分割、可独立重聚合。report 从它重聚合。

增量去重 & 帧去重

增量去重（scan 阶段，跨次运行）

用 (mtime, size) 而非内容哈希：NAS 上视频多，算 SHA 太慢；对“同名文件被覆盖”这类场景， mtime+size 已足够可靠，而内容哈希只在“文件原地编辑但大小时间不变”时才有优势——监控视频不会发生。

状态文件损坏时自动备份 .corrupt-*.json 并重建空状态。

帧去重（extract 阶段，单视频内）

相邻帧用 dHash 计算汉明距离，≤ dedup_hamming_threshold 则合并为一个代表帧， 覆盖 time_range。agent 只识别代表帧，ingest 按 time_range 把结果扩展回整段， 聚合逻辑无需感知去重。

事件聚合规则

把“逐帧识别结果”变成“行为事件”，实现见 scripts/aggregate.py：

  1. 跳过无效帧：idle/absent 不参与事件，但会打断连续段。
  2. 连续段切分：相同 activity.type 连续归入同段，类型变化开新段。
  3. 跨段间隔合并：两个同类型段，若后段首帧 − 前段末帧 ≤ event_merge_gap_seconds（默认 30s）合并。30 秒容忍窗口吸收“猫低头吃粮某帧被识别成 idle”的噪声，又不会把两次独立行为误并。
  4. 最短帧数过滤：合并后段帧数 < min_frames 丢弃——单帧活动太可能是误判。 ✅ 已实现：min_frames 按 context 分段，猫砂盆 = 1、喂食机 = 2（见下）。
  5. 参与猫的判定：事件内每帧 cats 投票，取出现频次最高的 identity；并列取置信度最高；全无则 unknown。
黑猫快速进出砂盆的系统性漏检 ✅ 已实现：黑猫有 < 24s 快速进出砂盆的行为。12s 采样间隔下，这种活动最多横跨 2 个采样点、极可能只命中 1 帧甚至 0 帧，被 min_frames=2 直接丢弃——每次都漏。解法（与采样间隔分段配套）：猫砂盆 context 采样间隔收到 6s（10 帧/段，<24s 活动能命中 3–4 帧），并把猫砂盆 min_frames 降到 1 兜底。预筛先砍掉 85% 无运动段，是 6s 间隔帧数不失控的前提。
为什么跨视频聚合：1 分钟视频是人为切片，猫的真实行为不关心文件边界。 run.py 把当天所有帧写入同一个 raw/YYYY-MM-DD.jsonl，聚合统一读取， 所以跨文件的“14:29:58 - 14:31:14 吃饭”能正确合并成一次。

核心原则

  1. token 由两个正交维度决定：送多少帧 × 每帧多大。预筛砍段数（最大杠杆）、降采样砍每帧 token（6×）、分段采样救黑猫漏检、dHash 去重省帧数——四者咬合。详见上方 token 消耗账。
  2. 绝不重复识别。scan 只返回未处理文件；ingest 处理完即标记；预筛的 silent 段也标记。这是定时启用的前提。
  3. 预筛不破坏数据。无运动段静默跳过、不写 jsonl（absent 帧本就不参与事件），但必须标记 silent 避免下次重列。
  4. 跨平台。路径走 pathlib，不硬编码挂载点，支持 macOS / Linux / WSL / Windows。
  5. 失败可重入。单个视频任何一步失败都不标记为已处理，下次 scan 会重试。
  6. 识别精度优先。宁可保守（过渡帧算 idle），也不要把噪声混进事件——聚合阶段 30 秒间隔容忍会补回短中断。

失败与重试

失败点行为
extract 失败不标记已处理；下次 scan 会重新列入 todo
识别时某帧判断不了agent 可对该帧填 idle/unknown，或整帧不写入 results
ingest 校验失败该条被拒（返回 errors），其余正常写入；视频整体仍标记已处理
状态文件损坏自动备份 .corrupt-*.json 并重建空状态
ffmpeg 缺失报错并提示安装方式（macOS brew / Linux apt / Windows winget）
