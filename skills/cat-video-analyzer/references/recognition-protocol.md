# 识别协议（Agent 必读）

本 skill 的逐帧识别**由你（agent）完成**，不是脚本调 API。本文是你识别每帧图片时
必须遵循的规则和输出格式。读图前先读这里。

## 你会拿到什么

`run extract`（或 `run batch-extract`）对视频抽帧并**去重**后输出一个 JSON 数组。
相邻、画面几乎相同的帧会被合并成一个**代表帧**，用 `time_range` 标明它覆盖的时段。
你只需识别代表帧，识别结果会被 `ingest` 自动扩展到整段。典型元素：

```json
{
  "frame_path": "/tmp/catva_xxx/frame_000001.jpg",
  "frame_ts": "2026-06-11T14:30:00+08:00",
  "context": "litter_box",
  "time_range": ["2026-06-11T14:30:00+08:00", "2026-06-11T14:30:24+08:00"],
  "represents": 3
}
```

- `frame_path`：代表帧图片路径。你用读图能力（Read / 多模态）打开它。
  代表帧已被 ffmpeg 降采样到长边 ≤ `frame_max_side`（默认 768px），单帧 token 比原生
  1080p 省 ~6x；768 足以看清虎斑条纹、是否低头进食等判据，无需担心细节丢失。
- `frame_ts`：代表帧的绝对时间（= 视频起始时间 + 帧偏移）。
- `context`：场景。**猫砂盆摄像头** → `litter_box`；**喂食机摄像头** → `feeder`。
  这个上下文对判断行为至关重要，不要忽略。
- `time_range`（可选）：该代表帧覆盖的时段 `[start, end]`。有此字段说明它代表了
  这段时间内多个相似帧——你的识别结果会被 `ingest` 扩展到整段。**回传时务必原样
  带上**。没有此字段则是普通单帧。
- `represents`（可选）：代表了几帧，仅供参考。

> 去重需 Pillow（`pip install pillow`）。未装时 `extract` 自动降级为输出全部帧
> （每帧都是普通单帧、无 `time_range`），流程不变，只是不省 token。`catva doctor`
> 会检查 Pillow 是否就绪。

## 要识别什么

对每张代表帧图，判断两件事：

### 1. 画面里有哪些猫

家里两只猫：
- **狸花猫 `tabby`**：棕色或灰色**虎斑条纹**，额头有 M 形花纹，身上有明显的条纹/斑点。
- **黑猫 `black`**：全身黑色，无明显花纹。

如果画面里有猫但无法确定是哪只（比如只露了尾巴、画面太暗），填 `unknown`。
没有猫就返回空数组 `[]`。

### 2. 猫在做什么

根据 `context` 判断：

| context | 算 `toileting` | 算 `eating` | 否则 |
|---------|----------------|------------|------|
| `litter_box`（猫砂盆） | 刨沙、蹲伏在砂盆内、进出砂盆 | **不可能**（强行判 `idle`） | `idle`（路过/休息）或 `absent`（无猫） |
| `feeder`（喂食机） | **不可能**（强行判 `idle`） | 低头、鼻尖/嘴靠近食碗或出粮口、咀嚼 | `idle` 或 `absent` |

**关键约束**：猫砂盆画面里不会出现真正的进食，喂食机画面里不会出现真正的如厕。
如果你在 `litter_box` 画面里看到猫低头，那更可能是刨沙或嗅闻，不是吃饭。
这个约束在 `ingest` 阶段还会做代码兜底，但你在识别时就应正确归类。

## 输出格式

对每帧（代表帧）产出一个 JSON 对象（**纯 JSON，不要 markdown 代码块、不要前后文**）：

```json
{
  "frame_ts": "2026-06-11T14:30:00+08:00",
  "context": "litter_box",
  "cats": [
    {"identity": "black", "confidence": 0.9}
  ],
  "activity": {
    "type": "toileting",
    "evidence": "黑猫蹲伏在砂盆中央，前爪正在刨动猫砂"
  },
  "time_range": ["2026-06-11T14:30:00+08:00", "2026-06-11T14:30:24+08:00"]
}
```

- `cats`：画面里每只猫一个元素。两只都在就两个。都没有就 `[]`。
- `activity.type`：`eating` / `toileting` / `idle` / `absent`。
  `absent` 仅当画面中完全没有猫。
- `activity.evidence`：**必须填**，用一句话引用画面细节。
  这不是走形式 —— 它逼你看图而非猜，也会写进最终报告的时间线供主人抽查。
  两只猫做不同的事时，在这里分别描述。
- `time_range`：**如果输入帧带了这个字段，原样回传**；`ingest` 会按采样间隔把结果
  扩展为整段内的多个时间点写回 jsonl，聚合照常工作。输入没有就别加。

## 批量识别（省往返）

去重后一个视频通常只剩 2-3 个代表帧。**建议一次打开该视频的全部代表帧、
一次性输出全部 JSON 对象（数组）**，而不是逐帧往返。图片 token 不变，但能省掉
每轮的协议/提示词重复开销。多个视频的代表帧也可在 `batch-extract` 后一起识别。

## 如何把结果写回

一个视频的所有代表帧都识别完后，把结果数组传给：

```bash
CATVA ingest \
  --video /path/to/clip.mp4 \
  --date 2026-06-11 \
  --results '[{"frame_ts":"...","context":"...","cats":[...],"activity":{...},"time_range":[...]}, ...]'
```

`ingest` 会：
1. 校验每个结果的字段合法性。
2. 应用路径级硬约束（猫砂盆目录的 `eating` → `idle`，反之亦然）。
3. **若结果带 `time_range`，按采样间隔扩展为整段内的多个时间点写入 `raw/日期.jsonl`**。
4. 把该视频标记为已处理（下次扫描跳过）。

## 常见陷阱

- **红外夜视画面**：猫可能只剩两个反光眼点。能判断行为就判断，判断不了就
  `idle`（有猫但状态不明）或保留 `unknown` identity。不要强行编造。
- **两只猫重叠/遮挡**：尽量分开判断；实在分不开，identity 填 `unknown`，
  evidence 注明"两只猫重叠难以区分"。
- **猫刚好吃完抬头**：这种过渡帧算 `idle`。只有明确的进食姿态才算 `eating`。
  宁可漏一帧（聚合时 30 秒间隔会补上），不要把 idle 混进 eating 污染统计。
- **代表帧覆盖时段内有变化**：理论上 `time_range` 内的帧都已被判定为"几乎相同"，
  所以扩展结果是安全的。如果你确实在代表帧里看到猫正处于**动作转换**（如正要离开），
  说明它不该被去重合并——可在 evidence 里注明，或把 `activity.type` 取整段的主导行为。
