# 事件聚合规则

把"逐帧识别结果"变成"行为事件"的规则。实现见 `scripts/aggregate.py`。

## 输入

一个按时间排序的 `FrameResult` 列表，每帧含：
- `frame_ts`：时间戳
- `activity.type`：`eating` / `toileting` / `idle` / `absent`
- `cats`：识别到的猫列表
- `context`：来自猫砂盆还是喂食机

## 规则

### 1. 跳过无效帧

`idle` 和 `absent` 帧不参与事件，但它们会**打断**连续段。

### 2. 连续段切分

遍历帧，相同 `activity.type` 连续 → 归入同一段；类型变化 → 开新段。
被 `idle`/`absent` 打断后，后续同类型帧开新段。

### 3. 跨段间隔合并

两个**同类型**段，如果后一段的首帧时间 − 前一段的末帧时间 ≤ `event_merge_gap_seconds`（默认 30s），合并为一段。

> 为什么允许间隔合并？识别会有抖动：猫低头吃粮时某帧被识别成 idle 很常见。
> 30 秒的容忍窗口能吸收这种噪声，同时不会把两次独立的行为（通常间隔数分钟）误并。

### 4. 最短帧数过滤

合并后的段如果帧数 < `min_frames`（默认 2），丢弃。
单帧的 `eating` 太可能是误判，不值得记录。

### 5. 参与猫的判定

事件内每帧的 `cats` 投票：
- 取**出现频次最高**的 identity 作为事件归属。
- 并列时取**置信度最高**的。
- 所有帧都没识别到猫 → `unknown`。

## 为什么跨视频聚合

1 分钟视频是人为切片，猫的真实行为不关心文件边界。例如：

```
文件A 14:29-14:30:  ...第58秒 eating(狸花)
文件B 14:30-14:31:  第2秒 eating(狸花), 第14秒 eating(狸花)...
```

如果只在文件内聚合，这会被拆成两个事件（一个 1 帧、一个 2 帧），1 帧的还会被
`min_frames` 过滤掉。`run.py` 把当天所有帧写入同一个 `raw/YYYY-MM-DD.jsonl`，
聚合时统一读取，所以跨文件的事件能正确合并成一次「14:29:58 - 14:31:14 吃饭」。

## 已知边界情况

- **两只猫同时吃饭**：当前只记录频次最高的一只。如需并行记录，需扩展事件结构
  为"每只猫独立事件流"。这是有意识的简化，避免双猫场景的归属歧义。
- **快速进出砂盆**（<24s，只命中 1 帧）：会被 `min_frames` 过滤。如果这类行为
  重要，调小 `frame_interval_seconds` 到 6-8 或把 `min_frames` 设为 1。
- **红外夜视纯黑画面**：模型常返回 `absent`，导致漏报。夜视场景建议换 Opus 或
  接受漏报（白天数据通常已足够反映进食/排泄规律）。
